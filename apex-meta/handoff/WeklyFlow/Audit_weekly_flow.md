**Core verdict:** The weekly value chain is real and mostly understandable. The repo-level loop explicitly defines “structured plans → Marco executes → execution evidence becomes durable project state” and lists the intended loop from PreCapWeek through next PreCapNextDay . The biggest problem is not doctrine. The biggest problem is that value is often visible only after reading contracts/templates, and several packages still expose path drift, old build residue, or schema-heavy presentation.

# 2. One-page user story walkthrough

**Marco starts the week... — PARTLY CLEAR**  
The root loop says APEX creates structured plans, Marco executes, and evidence becomes durable project state . That is the right story. But the loop itself is collapsed into a dense machine block, not a human walkthrough .

**Marco runs PreCapWeek because... — PARTLY CLEAR**  
PreCapWeek says it turns weekly intent, project state, status overview signals, calendar constraints, and the weekday blueprint into `precap_week_output` and `first_precap_next_day_seed` . Value is understandable, but the entrypoint uses collapsed YAML and points to support files that appear incomplete or path-fragile .

**Then he runs PreCapNextDay because... — CLEAR**  
PreCapNextDay has a strong value statement: compile partial planning/project/recap/calendar/prompt/workflow/usage context into `next_day_plan` artifacts without executing work or requiring complete inputs . Its contract and procedure make the transformation clear: load best context, select mode, represent flows, create flow packets and prompt packs, prepare calendar requests only as requests, validate, and surface review flags .

**After execution, he uses RawFlowDumpNormalize because... — CLEAR**  
This is one of the clearest packages. It says it turns messy post-execution evidence into either `normalized_raw_flow_dump` or `skipped_flow_marker` for FlowRecap, without running FlowRecap or mutating state . The template is operator-friendly: flow card, source refs, raw evidence, normalized interpretation, model usage notes, confidence, and gaps .

**FlowRecap helps him... — CLEAR**  
FlowRecap turns one completed/partial/skipped/blocked flow plus normalized raw evidence into a reviewable recap packet with candidate-only project status and model usage deltas . Its template makes candidate-only boundaries highly visible, especially for project status and model usage deltas .

**ModelUsageLog helps him... — PARTLY CLEAR**  
The practical value is good: learn from actual model/surface usage and produce reuse/avoid signals for later planning . The output is useful but still a bit schema-heavy; it needs a stronger “what did we learn?” card.

**StatusMerge helps him... — CLEAR**  
StatusMerge’s value is explicit: review FlowRecap-derived candidate deltas, compare them to prior state, surface conflicts, produce a proposal-only merge packet, and prepare `next_PreCapNextDay_input_context` without mutating durable KB state . Its template is very usable because it puts operator review and conflicts first .

**project-kb-manager helps him... — PARTLY CLEAR**  
The package says it maintains the Apex project KB as compact schema-governed project status, milestone state, FlowRecap consumption, and PreCapNextDay context . But the newer orchestration-state packet appears in the manifest and template , while the SKILL entrypoint’s supporting-files list does not yet expose that new packet .

**The next PreCapNextDay becomes better because... — CLEAR IN STATUSMERGE / PARTLY CLEAR IN KB**  
StatusMerge has an explicit next PreCap context section with focus, ready actions, blockers, decisions, usage refs, confidence, and validation status . project-kb-manager also now has a next-day planning view template for PreCapNextDay , but the entrypoint should surface it.
