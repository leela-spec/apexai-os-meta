## Essential design requirements

Preserve these fundamentals:

1. **Operator value first**  
    The operator should quickly understand what is ready or changed.
    
2. **Next action is obvious**  
    Every primary template should make the best next action visible.
    
3. **Review and uncertainty are visible**  
    Important assumptions, conflicts, low confidence, and review needs must not be hidden.
    
4. **Human-readable before machine-readable**  
    Operator comprehension is primary. Structured handoffs are secondary and compact.
    
5. **No invented domain meaning**  
    Templates project existing contracts; they do not create replacement schemas.
    
6. **One dominant job per artifact**  
    Shared visual language must not erase artifact-specific purpose.
    
7. **Link rather than duplicate**  
    Related artifacts should reference each other instead of repeating full content.
    
8. **No silent state transition**  
    Candidate, approved, written, confirmed, and recommended states must remain visibly distinct.
    
9. **Practical usability**  
    Another capable AI should be able to fill each template correctly without guessing its purpose.
    
10. **Avoid overengineering**  
    Use only the structure that improves operator comprehension, action, review, or downstream handoff.

## Important artifact boundaries

Pay particular attention to these boundaries:

### J3 and J4

J3 is the compact next-day, three-sprint outline.

J4 is the complete single-flow execution workspace.

J3 should summarize and link to J4 rather than repeat:

- complete tasks;
    
- complete inputs;
    
- full dependencies;
    
- done conditions;
    
- stop or review conditions;
    
- full prompt content.
    

### J4 and J5

J4 owns execution context.

J5 owns direct prompt-file access and prompt indexing.

Do not embed complete prompt content into J4 or repeat the complete execution plan in J5.

### J6 and J7

J6 organizes evidence.

J7 interprets what the evidence means.

Evidence collection must not silently become a project-state conclusion.

### J7, J9, J10, and J11

Keep these concepts separate:

```text
J7  → candidate project-state change
J9  → operator merge decision
J10 → durable KB write and its result
J11 → confirmed accepted project truth
```

Approval is not proof that writing occurred.

A prepared update is not a confirmed durable result.

J11 should not present new truth unless the durable update is confirmed.

### J8 and J12

J8 provides advisory usage learning.

J12 produces an advisory route recommendation and records the operator route decision.

Do not turn usage learning into an automatic route override.

### J12 and execution

A recommended route does not authorize execution by itself.
