# Synthesis Verdict

Proceed to patch planning, but **do not implement the union of R01–R04**.

The coherent minimum is:

1. evidence-bound completion;
2. capability-routed execution;
3. one deterministic postflight evidence packet;
4. stronger reason-coded deterministic quality detection;
5. one bounded clean-context semantic acceptance gate;
6. targeted repair only when either gate fails;
7. consolidation to one lifecycle authority.

This directly addresses the named root-cause failures: deterministic quality was not executed, connector read-back replaced validation, the quality heuristic missed heading-complete thin pages, artifact count displaced usefulness, and the specified repair loop did not run. R01 frames these as executor and completion-state failures; R02 reproduces the deterministic false negative; R03 demonstrates the semantic defects on actual pages; R04 shows that duplicated process authority did not enforce any of those controls.

# Where All Four Reports Agree

## 1. Structural compliance is not semantic quality

All four reject the substitution of file existence, headings, frontmatter, source lists, or connector read-back for useful compiled knowledge.

- R01 identifies headings, file existence, read-back, and documented commands as false completion proxies.
- R02 reproduces the exact deterministic false-negative path: all headings and references can exist while one generic claim and file-level pointers still pass.
- R03 semantically inspects those pages and finds unsupported scope, mislabeled inference, weak synthesis, and poor query utility.
- R04 identifies the same proxies as the central process failure.

**Agreement class:** strong convergence, but not four independent datasets. Every report ultimately relies on the same max-run root-cause evidence and overlapping page sample. The analytical methods are partly independent—runtime inspection, metric analysis, semantic entailment review, and process audit—but the underlying incident is shared.

## 2. Deterministic and semantic validation must remain separate

R02 explicitly limits deterministic authority to measurable structure. R03 requires a semantic gate for claim support and query usefulness. R01 makes deterministic validation and semantic acceptance separate transitions. R04 makes the same separation central to its simplified lifecycle.

**Agreement class:** partially independent. R02 derives the boundary from what Python can measure; R03 derives it from source-entailment failures; R01 derives it from executor capabilities; R04 derives it from process simplification.

## 3. No universal word threshold or aggregate page-quality score

All reports reject a universal length floor, holistic numeric quality score, or prose-quality grading. Valid narrow pages can be concise, and aggregate scores can hide blocking grounding failures.

**Agreement class:** mostly shared design principle, with limited empirical support from R02 and R03’s concise entity-page sample.

## 4. Failures should produce reason-coded, targeted repair

R01 requires a reason-coded repair set. R02 proposes machine-readable candidate objects with measurements and reason codes. R03 requires evidence-backed findings and targeted re-review. R04 rejects blind rewrites and makes candidate-driven repair part of its minimum process.

**Agreement class:** shared recommendation tied directly to the observed missing repair loop, but not yet validated through a completed repair cycle.

## 5. Semantic review must be bounded

All four reject a second full drafting pass, review of every sentence, or operator review of every page. The gate should focus on target queries, material claims, and blocking defects.

# Genuine Conflicts and Decisions

## Conflict 1: Five stages versus seven stages

R01 proposes five operational stages. R04 proposes seven target-facing stages by treating candidate repair and truthful completion as stages. R03 describes a sequence closer to five stages. R02 does not propose a lifecycle.

**Decision:** R01 wins on operational stage count.

Candidate repair is a conditional loop, not a normal stage. Truthful completion is a derived result, not an activity. Capability checking is a transition precondition rather than a separate process phase.

The synthesis therefore uses **five operational stages**, with repair as a loop and completion as an evidence-derived state.

## Conflict 2: Meaning of `validated`

R01 uses:

- `compiled_unvalidated`;
- `deterministically_validated`;
- `query_ready` after semantic acceptance.

R03 uses `validated` to mean deterministic checks plus no semantic failure. R04 prefers only three externally reportable classes: partial, failed, complete.

**Decision:** use two layers.

Internal evidence fields:

```
compiled: true|false
deterministic_postflight: pass|fail|unrun
semantic_acceptance: pass|partial|fail|unrun|insufficient_evidence
retrieval_freshness: fresh|stale|missing|unknown
```

Externally report only:

```
partial
failed
complete
```

`query_ready` is permitted only as a capability assertion within `complete`, not as a loosely selected success label.

R04 wins on external simplicity. R01 wins on preserving distinct internal evidence milestones. R03’s overloaded use of `validated` should not be adopted.

## Conflict 3: Keep or delete named output tiers

R01 recommends retaining output tiers with stronger evidence definitions. R04 recommends deleting or reducing them unless runtime behavior consumes them. R02 and R03 do not depend on them.

**Decision:** do not delete output tiers in this patch.

Output tiers were not a named cause of the observed max-run failure. Removing them now would broaden the patch without directly fixing postflight omission, proxy completion, or semantic thinness.

They may be reconsidered later, but are rejected from the immediate patch scope under the observed-failure rule.

## Conflict 4: How strict deterministic thinness checks should be

R02 proposes calibrated page-type thresholds, including claim-count and section-length candidate rules. R04 says thinness should remain semantic unless thresholds have accepted fixtures. R03 demonstrates that brevity alone is not failure and that a narrow page can be valid.

**Decision:** R04 and R03 win on blocking policy; R02 wins on measurement.

Implement the measurements immediately, but:

- block only objective structural failures;
- emit thinness patterns as reason-coded candidates;
- promote numeric thresholds to strict failures only after calibration against operator-accepted fixtures.

## Conflict 5: Must the semantic evaluator be a different model?

R01 describes an independent reviewer that did not draft the pages. R03 permits the same model in a clean, separate invocation. R04 likewise treats these as capability functions rather than permanent model identities.

**Decision:** R03 wins.

The default is a **clean-context separate evaluation invocation**. A different model or operator review is escalation for high-impact pages, disputes, or ambiguous evidence. Model identity alone does not establish independence.

## Conflict 6: One postflight packet versus a new aggregate command

R04 treats one postflight packet as mandatory. R01 treats a new aggregate postflight command as optional.

**Decision:** the packet is mandatory; a new command is optional.

Existing commands may initially produce the packet through orchestration. Patch planning must not assume that a new workflow engine inside `apex_kb.py` is required.

# Mandatory Changes

## 1. Evidence-derived completion and connector state cap

**Observed failures prevented:**

- connector read-back represented as validation;
- final completion label overstated what was proven;
- future terminal validation was treated as a qualified success.

Required behavior:

```
connector_write_only:
  maximum_state: compiled_unvalidated
  external_result: partial
  forbidden_claims:
    - validated
    - postflight_complete
    - quality_validated
    - query_ready
    - complete
```

R01 and R04 both make this the highest-confidence process correction.

## 2. Capability routing before entering a stage

**Observed failure prevented:** the active executor lacked terminal execution but started and reported a workflow whose completion required it.

The process must route by capability, not by product name. Missing capability must cap the state and emit one bounded handover; it must not weaken the completion criteria.

## 3. Mandatory deterministic postflight evidence packet

**Observed failure prevented:** index, retrieval rebuild, stale check, lint, quality, and status were never executed after semantic writes.

The packet must identify the exact tested commit or tree and include:

```
required_results:
  - changed_path_scope
  - source_custody_status
  - index_result
  - retrieval_build_result
  - retrieval_stale_result
  - lint_strict_result
  - quality_strict_result
  - status_result
```

A new aggregate CLI command is not mandatory for the first patch, but omission of any required result blocks completion.

## 4. Strengthen deterministic quality using reason-coded measurements

**Observed failure prevented:** pages with all headings, references, and one generic claim escaped strict quality.

Mandatory measurements:

- per-section narrative counts;
- key-claim count;
- claim-to-pointer coverage;
- pointer-specificity distribution.

These four directly address the observed one-claim, one-sentence-layer, file-pointer-only failure. Ranked-source, route, uncertainty, and total-word counts may be emitted but should not independently block completion.

The quality command must return structured candidate objects:

```
path: wiki/...
page_type: concept
reasons:
  - SINGLE_CLAIM_CONCEPT_THIN
  - ALL_SYNTHESIS_LAYERS_SHORT
  - CLAIM_POINTER_FILE_ONLY
measurements: {}
threshold_status: experimental
```

## 5. Bounded clean-context semantic acceptance

**Observed failure prevented:** deterministic structure passed while pages remained weakly grounded and query-useless.

Minimum acceptance:

1. test at least one declared target query;
2. select two or three material claims independently;
3. resolve and inspect cited passages;
4. check inference labels, uncertainty, and scope;
5. return `pass`, `partial`, `fail`, or `insufficient_evidence` with page and source pointers.

The evaluator must not see drafting rationale, desired verdict, or self-reported completion evidence.

## 6. Candidate-driven repair loop

**Observed failure prevented:** the specified existing-page repair flow did not run.

Only affected pages or deterministic artifacts are repaired. After repair:

- rerun deterministic checks for changed artifacts;
- rerun semantic review only for repaired blocking findings;
- do not regenerate the whole KB by default.

## 7. One lifecycle authority

**Observed failure prevented:** the process existed in prose but did not govern execution, while multiple surfaces offered different gate and completion interpretations.

`SKILL.md` becomes the sole lifecycle and completion authority. Other files are restricted to runtime behavior, tests, inventory, or examples.

# Rejected Recommendations

## Rejected from the immediate patch

- **Delete named output tiers now.** R04 proposes it, but it does not directly prevent the documented quality or validation failure.
- **Mandatory separate Phase 1→2 approval for every compile.** It did not prevent the observed incident and can block an explicitly authorized full compile.
- **Universal claim-count or section-word failures.** They risk invalidating concise entities and remain uncalibrated.
- **Strict failure solely because all pointers are file-level.** R02 and R03 both recognize valid concise or source-format exceptions.
- **Strict zero-route or placeholder detection as the core quality repair.** These may improve fixtures, but they do not explain the observed heading-complete page failure and therefore are not part of the minimum fix.
- **Mandatory different-model review.** Different model identity does not guarantee correctness.
- **Review every page or every sentence.** This turns the semantic gate into a second compile.
- **A permanent multi-agent hierarchy or named executor roles.**
- **A new lifecycle document.**
- **Mandatory branches or pull requests for every KB operation.**
- **Graph extraction as a blocking quality gate.**
- **Query-eval schema validity as semantic usefulness proof.**
- **A universal page-value score.**
- **A universal word minimum.**
- **Exhaustive corpus reading.**
- **Embedding full Git workflow orchestration inside `apex_kb.py`.**
- **Rollback by default after any failure.**

# Smallest Viable Process

1. **Source custody and deterministic preparation**  
    Resolve requested outputs and executor capabilities; preserve sources; run Phase 0 only when needed.
2. **Semantic compile and persistence**  
    Produce the requested analysis/wiki artifacts with bounded source coverage and pointers. The resulting state is `compiled_unvalidated`.
3. **Deterministic postflight**  
    A terminal-capable executor validates the exact artifact set and produces one postflight evidence packet.
4. **Clean-context semantic acceptance**  
    Test target queries and two or three material claims against resolved evidence.
5. **Targeted repair or completion**  
    Repair only reason-coded failures and rerun affected checks. Otherwise derive `complete`; fresh retrieval permits `query_ready`.

Capability routing is a precondition on each stage. Truthful completion is a derived result, not an additional stage.

    Always persisted, or required only for auditability, contradiction handling, or operator review.

    Whether `complete` alone is reported externally or whether `query_ready` remains a visible subtype of complete.
