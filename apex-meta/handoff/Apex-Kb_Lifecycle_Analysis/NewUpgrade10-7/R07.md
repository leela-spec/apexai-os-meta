research_report:  
id: R07

recommendation: ACCEPT

binary_decision: >  
Add one thin aggregate `postflight` command to `apex_kb.py`.

material_reduction_in_terminal_execution_errors: true

conclusion: >  
A fixed aggregate command would materially reduce skipped stages, incorrect  
ordering, stale-index validation, inconsistent flags, and false completion.  
It does not become a workflow engine if it delegates to existing functions,  
has no persisted state, introduces no retries or dynamic routing, and adds no  
duplicate validation logic.

exact_evidence:  
existing_interfaces:  
lifecycle_index:  
implementation: cmd_index  
behavior: Rebuilds the machine section of wiki/index.md.  
write_policy: Existing dry-run and allow-write policy.

```
  lifecycle_lint:
    implementation: cmd_lint
    behavior: >
      Runs path, frontmatter, wikilink, payload-manifest, value-contract,
      and wiki-index freshness checks.
    strict_behavior: >
      Returns fail only when blocking findings exist and strict mode is
      enabled.

  lifecycle_quality:
    implementation: cmd_quality
    behavior: >
      Produces source/page maps, reason-coded repair candidates, shell
      candidates, and page metrics.
    strict_behavior: >
      Returns fail whenever phase2_repair_candidates or shell_page_candidates
      are non-empty.

  lifecycle_audit:
    implementation: cmd_audit
    behavior: Lists and groups audit items without mutation.

  lifecycle_status:
    implementation: cmd_status
    behavior: >
      Reports wiki index, retrieval index, source payload manifest, Phase 0,
      source, wiki-page, and audit status.

  retrieval_build:
    implementation: apex_kb_retrieval.cmd_build_index
    behavior: >
      Builds JSON, NDJSON, metadata, and SQLite FTS5 when available. Existing
      write helpers enforce kb-root boundaries and dry-run behavior.

  retrieval_stale:
    implementation: apex_kb_retrieval.cmd_stale
    behavior: >
      Compares current wiki page paths and hashes against index metadata and
      reports added, deleted, and modified pages.

dispatch_evidence:
  - >
    `apex_kb.py` already dispatches each lifecycle command through one
    function and maps fail, blocked, and error statuses to nonzero exit.
  - >
    `apex_kb_retrieval.py` exposes build-index and stale as callable Python
    functions, so aggregation does not require shelling out.
  - >
    Existing `cmd_query` already imports the retrieval module internally,
    establishing an existing delegation pattern.
  - >
    Every write-producing function already receives allow_write and derived
    dry_run state; aggregation can pass through the same Namespace values.
  - >
    Current documentation requires several separate commands, creating a
    human sequencing surface not protected by one terminal exit result.
```

comparison:  
existing_multi_command_postflight:  
decision: reject_as_primary_interface  
strengths:  
- no implementation change  
- each command independently inspectable  
weaknesses:  
- commands can be skipped  
- commands can be run out of order  
- different kb-root or flag placement can be used accidentally  
- later stages may validate stale pre-rebuild artifacts  
- no single machine-readable verdict  
- shell chaining differs across PowerShell, cmd, Bash, and CI  
- interruption can leave uncertain evidence  
- operators can mistake partial command success for complete postflight

```
thin_orchestration_command:
  decision: accept
  strengths:
    - fixed stage order
    - one kb-root and one resolved mode
    - one exit status
    - one JSON evidence envelope
    - delegates existing implementations
    - can stop dependent stages after a build failure
    - preserves stage-level outputs for diagnosis
  constraints:
    - no validation logic duplicated
    - no new lifecycle state
    - no retries
    - no persisted orchestration ledger
    - no dynamic stage graph
    - no semantic review
    - no retrieval ranking changes
    - no shell subprocess calls

documentation_only_sequence:
  decision: reject
  reason: >
    Documentation already describes multi-command sequencing. It cannot
    enforce execution, shared arguments, dependency order, interruption
    behavior, or a combined terminal verdict.
```

minimum_command_contract:  
command: postflight

```
fixed_stage_order:
  - stage: wiki_index
    delegate: cmd_index
    required_write_in_execution_mode: true

  - stage: retrieval_build
    delegate: apex_kb_retrieval.cmd_build_index
    required_write_in_execution_mode: true

  - stage: lint
    delegate: cmd_lint
    forced_options:
      strict: true

  - stage: quality
    delegate: cmd_quality
    forced_options:
      strict: true

  - stage: audit
    delegate: cmd_audit
    blocking: false
    purpose: evidence_only

  - stage: status
    delegate: cmd_status
    blocking: true

  - stage: retrieval_stale
    delegate: apex_kb_retrieval.cmd_stale
    blocking: true
    required_final_status: fresh

excluded:
  - query-eval
  - graph
  - semantic review
  - source intake
  - Phase 1
  - Phase 2 drafting
  - commit
  - push
  - retrieval ranking changes

exclusion_rationale: >
  Those are separate lifecycle or optional validation surfaces. Adding them
  would turn a bounded deterministic postflight into a larger workflow.
```

dry_run_contract:  
default: true  
behavior:  
- resolve the complete fixed stage plan  
- invoke write-producing delegates in their existing dry-run mode  
- preserve their planned-write evidence  
- run independent read-only checks against current repository state  
- do not claim postflight pass  
- do not claim rebuilt or fresh artifacts based on planned writes  
final_status: planned  
evidence_complete: false  
writes_performed: false

allow_write_contract:  
enabled_by: --allow-write  
behavior:  
- execute index rebuild  
- execute retrieval rebuild  
- run all required read-only checks against the resulting files  
- return pass only when all blocking stages pass and retrieval is fresh  
dry_run_override: >  
`--dry-run` continues to override `--allow-write`, matching existing  
effective_dry_run behavior.

failure_behavior:  
stage_exception:  
action:  
- record stage status error  
- stop stages that depend on that stage  
- return overall status error  
- preserve prior stage evidence  
masking_forbidden: true

```

  skip:

    - quality
```

acceptance_tests:  
- >  
Default invocation returns planned, writes nothing, and sets  
evidence_complete false.  
- >  
`--allow-write` rebuilds index and retrieval artifacts before stale and  
status acceptance checks.  
- >  
`--allow-write --dry-run` writes nothing and returns planned.  
- >  
A quality repair candidate makes overall status fail and produces nonzero  
exit while retaining all quality reasons.  
- >  
A lint strict failure makes overall status fail.  
- >  
A retrieval build exception is not converted into stale pass.  
- >  
Missing or unreadable retrieval metadata cannot produce pass.  
- >  
Every stage receives the same resolved kb-root.  
- >  
No file outside kb-root is written.  
- >  
Direct individual commands remain supported without behavior changes.

unresolved_questions:  
- >  
Whether wiki-index failure should prevent retrieval build entirely or  
permit retrieval over the current wiki pages for diagnostic purposes.  
Recommended default: stop retrieval build because postflight evidence  
should represent one coherent rebuilt state.  
- >  
Whether audit findings should ever block. Current audit output lacks a  
severity contract, so blocking would invent new logic and is rejected.  
- >  
Whether query-eval should later become an optional flag. It should not be  
in the minimum command until its role in mandatory postflight is explicitly  
established.
