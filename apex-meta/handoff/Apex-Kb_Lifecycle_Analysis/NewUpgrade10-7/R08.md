minimal_capability_check:  
placement: >  
Immediately after the SKILL.md title and before the operating contract,  
lifecycle overview, file-navigation table, or procedure.  
decision_question: >  
Can the active executor run the repository's Python commands in a live  
worktree and capture command exit status plus stdout/stderr?  
reason_for_single_question: >  
Terminal Python execution with captured evidence is the capability that  
separates semantic authoring from deterministic lifecycle execution.  
Separate questions about retrieval, lint, quality, status, hashes, and  
indexes are derivative and need not be exposed to a connector executor.  
routes:  
terminal_backed:  
condition: >  
The executor can run repository commands in the live worktree and  
retain their outputs and exit status as evidence.  
route_to: terminal_route  
connector_only:  
condition: >  
The executor cannot run repository commands, but can read source files,  
create or fully rewrite complete semantic files, and re-read the exact  
committed or written content.  
route_to: connector_route  
unsupported:  
condition: >  
The executor can neither execute terminal commands nor reliably write  
and read back complete semantic files.  
outcome: blocked  
required_record:  
execution_surface: terminal_backed | connector_only | unsupported  
explicitly_not_required:  
- independent capability questions for every Python subcommand  
- inference from tool names or product identity  
- reading command contracts before choosing a route  
- attempting a command to discover that no terminal exists
