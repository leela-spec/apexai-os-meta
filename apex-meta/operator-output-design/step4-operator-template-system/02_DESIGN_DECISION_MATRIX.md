# APEX Template System - Design Decision Matrix

Evidence identifiers are defined in [Research Findings](01_RESEARCH_FINDINGS.md). Rows cover material system choices; minor typography and wording decisions are intentionally omitted.

| Decision | Problem | Options considered | Evidence | Selected approach | Why selected | Trade-offs | Applied to |
|---|---|---|---|---|---|---|---|
| Shared anatomy | Different jobs need recognition without semantic collapse | Identical schema; unrelated files; shared presentation layers | R01 R02 R04 U05 | Shared result/action/review/provenance/handoff anatomy; artifact-specific bodies | Supports fast recognition while preserving ownership | Some repetition is deliberate; maintainers must resist adding universal states | J1-J12 except J6a scale |
| First-screen contract | Operator must know value and action quickly | Metadata first; narrative intro; compact result card | R01 R02 U01 W05 | Four-line result card first | Directly answers state, outcome, next action, and review | Requires disciplined summaries; cannot carry all nuance | J1-J12 |
| Skip Marker exception | A tiny non-execution record would be overbuilt by the shared card | Full card; no record; one-to-three-line marker | R10 | No result card; minimal reason and handling | Preserves downstream meaning at lowest cost | Less context; escalates to J6 when evidence exists | J6a |
| Result versus validation | Build status can obscure operator value | Validation banner first; result first; merged view | R01 P03 | Operator outcome first; validation last unless it changes action | Keeps technical assurance subordinate to work value | Auditors must scroll or follow references | All |
| Action placement | Operators lose time searching for the next gate | Actions at end; actions in every section; one dominant action near top | R01 R02 U01 W05 | One best next action in result card; detailed valid actions immediately after | Improves invocation and interruption recovery | A complex artifact may have secondary actions later | All primary |
| Action vocabulary | Generic approve can mean planning, merge, write, or route | Generic verbs; lifecycle-specific verbs | R02 R04 | Use artifact-specific phrases such as approve week, approve for merge, confirm write, approve route | Prevents silent authority transfer | Longer labels | All |
| Result-card format | Markdown needs a portable visual summary | HTML cards; tables; blockquote; YAML front matter | W14 R02 | Plain blockquote with bold labels | Readable in source and rendered views; no CSS dependency | Less visual styling than a web component | J1-J12 |
| Heading hierarchy | Dense artifacts need navigable structure | Flat bullets; arbitrary headings; logical H1/H2/H3 | W06 W14 | One H1 and sequential descriptive headings | Improves scanability and assistive navigation | Requires template fillers to preserve hierarchy | All |
| Progressive disclosure | Background and provenance can overwhelm decisions | Everything expanded; hidden critical content; layered sections | R01 R02 W09 | Keep decision/action visible; move provenance, handoff, validation later | Reduces cognitive load without hiding essentials | Long documents still require scrolling for detail | All |
| Warnings | Frequent warnings create alert fatigue | Warn on every uncertainty; no warnings; material-only callout | R02 W10 | Show warning only when it changes decision or next action | Preserves salience | Minor issues remain in review flags | All |
| Review flags | A label alone does not tell the operator what to do | Flag only; issue/impact/action triad | R02 W05 | Each material flag states issue, why it matters, and required action | Turns uncertainty into a resolvable task | Adds a few lines | All |
| Confidence | Constant confidence labels add noise; hidden uncertainty harms trust | Always show; never show; conditional | R01 R02 R18 W13 | Show when below high or materially decision-relevant | Balances trust and density | High-confidence claims may lack a visible score | All |
| Provenance density | Trust needs sources, but full inventories overload | No provenance; full inventory; compact decisive refs | R01 R02 W13 | Main refs, freshness, assumptions, and gaps; link full evidence elsewhere | Supports verification with low clutter | Depends on stable references | All |
| Placeholder syntax | Fillers need unmistakable blanks that remain machine-safe | Angle brackets; prose instructions; Mustache placeholders | P04 W14 | Quoted `{{UPPER_SNAKE_CASE}}` in YAML and visible text | Easy to search and replace; low collision with Markdown | Could conflict with template engines if imported unchanged | All |
| Empty values | Blank fields can be mistaken for missing work or certainty | Leave blank; delete; explicit None/Unknown | R01 R02 W05 | Use explicit `None`, `Unknown`, or `Not verified`; remove inapplicable optional sections | Distinguishes absence from omission | Slightly more verbose | All |
| Optional sections | Rigid forms accumulate empty bureaucracy | All mandatory; implicit optionality; labeled optionality | P04 P05 | Headings say include when material; remove when unused | Teaches correct compression | Requires editorial judgment | All |
| Repeatable sections | Project and flow counts vary | Fixed slots; tables; repeatable blocks | R03 U02 | Mark repeat per project/flow/item | Scales without arbitrary caps | Long portfolios can still be lengthy | J1 J2 J3 J6 J11 |
| Cards versus lists | Modular records and sequences have different reading patterns | Cards everywhere; lists everywhere; purpose-based mix | W11 W12 W08 | Heading-based cards for modular entities; lists for sequence and actions | Matches information relationships | Slightly less uniform appearance | All |
| Tables | Wide tables harm Markdown scanability but comparisons sometimes need them | No tables; tables default; narrow conditional tables | R01 W12 P07 | Use only for comparable compact rows; minimize columns | Retains comparison power without dashboard overload | Some mobile viewers still scroll | J5 J11; decision matrix |
| Machine-readable layer | Downstream AI needs stable state without a second schema | No handoff; full contract copy; minimal projection | R02 P04 P05 | Compact YAML `presentation_handoff` at end | Preserves identity, state, refs, review status | Not sufficient as a domain packet by itself | J1-J12 |
| Authority metadata | Implementation must expose design and contract basis | Front matter; omitted refs; final static block | R17 | Static `template_authority` YAML last | Auditable without blocking first screen | Adds implementation detail to every file | All templates |
| Schema ownership | A complete-looking template can accidentally redefine contracts | Copy schema; invent missing fields; reference owner | R00 R01 P05 | Reference owning contracts and project only operator-visible minimum | Prevents drift and unauthorized domain change | Fillers may need to open the contract for full packets | All |
| Internal links | Duplication grows when artifacts cannot be opened directly | Copy content; opaque IDs; descriptive relative links plus path | R07 R08 W07 | Descriptive Markdown link and visible relative path fallback | Supports one-click access and plain-text recovery | Paths must be maintained when files move | J3-J6 and all refs |
| J1 detail | Project truth can become a database dump | Full KB; one-line summary; compact per-project state card | R03 U02 | Portfolio summary plus repeatable project cards and freshness | Enough to plan without re-reading the KB | Historical detail stays linked | J1 |
| J2 scope | Weekly plans need variable project depth | Three global priorities; exhaustive task database; per-project outcome blocks | R03 U02 | Portfolio direction plus repeatable project sections and cross-project sequence | Avoids arbitrary caps while remaining navigable | Large weeks produce longer briefs | J2 |
| J3/J4 depth | Day summary and workspace were previously conflated | Duplicate full detail; summary only; linked split | R03 R17 | J3 concise S1-S3 outline; J4 full execution detail | Makes day scannable and each flow executable | Requires reliable J4 links | J3 J4 |
| J4 resume point | Interruption makes long workspaces expensive to re-enter | Infer from notes; global next action; sprint-specific resume line | U03 R07 | Start or resume here plus sprint status | Reduces re-reading and context loss | Must be updated during execution | J4 |
| Prompt access | Large prompt packs repeat execution context | Embed prompts in J4; monolithic pack; one file per prompt plus index | R08 P05 W03 | Direct prompt files and lightweight index | Prompts are copyable, testable, and independently versionable | More small files in actual use | J5 with J4 links |
| Exact prompt content | Prompt body is needed, routing doctrine is not | Only link; include full plan; prompt file skeleton | R08 W01 W02 | Prompt file holds title, surface, use-when, expected return, body, routing ref | Makes execution direct without duplicating J4 | Prompt quality still depends on Prompt Engineering | J5 |
| Evidence routing | Normalization should not become mandatory ceremony | Always normalize; never normalize; criteria-based route | R09 | Direct, normalize, or insufficient based on evidence quality | Uses lightest reliable handoff | Requires judgment at the boundary | J6 |
| Evidence/interpretation split | Organized facts can silently become conclusions | Combined recap; strict separate artifacts | R09 R18 W13 | J6 facts/readiness only; J7 interpretation/candidates | Protects provenance and reviewability | Adds a handoff for complex runs | J6 J7 |
| Candidate notice | Operators may read proposed status as accepted | Small label; prominent notice; separate file | R18 R04 | Prominent candidate-only line before delta | Prevents accidental durable assumption | Adds repeated boundary text | J7 |
| Usage signal | Usage learning can become an unsupported benchmark | Free-form advice; numeric score; one accepted contextual signal | R11 | Exactly one accepted route signal plus context and evidence | Comparable and bounded | Cannot express every nuance in the signal alone | J8 |
| Cost/quota claims | Provider data changes quickly | Always include; omit entirely; include only when sourced | R11 R16 W13 | Optional sourced resource note with date | Avoids stale operational truth | May omit useful but unverified intuition | J8 J12 |
| J9 approval | Approval can be mistaken for completed merge | Single merged checkbox; decision plus confirmation | R13 R04 | Separate merge decision from durable write confirmation | Preserves operator authority and factual write status | Two-stage completion is less visually simple | J9 |
| J10 write states | Preparation and execution have different consequences | Prepared/executed binary; detailed runtime log; accepted state set | R14 R19 | Expose prepared, awaiting confirmation, executed, partial, skipped, blocked and result ref | Makes actual durable effect auditable | More state labels in this artifact only | J10 |
| J11 truth gate | Overview can amplify unconfirmed changes | Show latest candidates; show accepted only; mixed lanes | R15 R04 | Confirmed accepted state only; stale/unconfirmed as review flags | Overview remains trustworthy | Newest proposal may not appear until write completes | J11 |
| J11 hierarchy | Older workstream layer conflicts with live contract | Project/workstream/task; project/task/subtask | R15 R20 | Project -> task -> subtask; workstreams referenced elsewhere | Aligns live ProjectStatus contract | Some strategic grouping is external | J11 |
| J12 stable surface | Exact model names and limits are volatile | Exact model always; abstract only; stable class plus verified exact option | R16 R21 W01 | Stable surface class default; exact model optional with source/date/window | Keeps route durable and current | Less immediately specific when no verification exists | J12 |
| Recommendation/decision split | AI recommendation can look like authorization | One route field; separate recommendation and operator decision | R16 R04 W05 | Record advisory recommendation, alternatives, then explicit operator decision | Supports correction, override, and governance | Adds a second status line | J12 |
| Example placement | Inline examples can be mistaken for required fields or truth | Examples in every template; no examples; one separate document | P04 P09 | Shared, clearly fictionalized example-fragments file with anchors | Keeps production templates blank and package learnable | Reader must follow a link | All |
| Example domain | Older Hermes workflow architecture is not current authority | Adopt full old architecture; ignore source; use work content only | P09 R00 | Use workflow content and risks, not profile/Kanban/cron authority | Provides realistic stress tests without architecture drift | Examples require careful disclaimers | Examples |
| Validation visibility | A validation checklist can dominate the operator interface | Validation first; hidden; end/conditional | R01 R17 | Technical validation in final report and last/conditional template section | Assurance exists without displacing action | Problems may be found after reading | All |
| Maximum useful complexity | Completeness pressure creates bureaucratic fields | Full schema projection; minimal note; job-bounded depth | R01 P05 U01 | Only structure that aids comprehension, action, review, or handoff | Human-first and maintainable | Some domain detail remains in contracts | All |
| Markdown portability | Custom HTML/CSS harms reuse | Rich HTML; renderer-specific syntax; CommonMark subset | W14 W15 | Headings, blockquotes, lists, links, narrow tables, fenced YAML | Works in source, GitHub, and common viewers | Less visual control | All |
| Accessible links | Opaque links slow scanning and assistive use | Click here; raw path only; descriptive label plus path | W07 R07 | Action and destination in link text; visible path fallback | Purpose is clear in context | Longer link labels | All |
| Change consequences | Operators need to know what approval will do | Action only; consequence line; long policy | W05 R04 | Short consequence statement at J9/J10/J12 gates | Supports informed control | Requires accurate maintenance | J9 J10 J12 |
| Source gaps | Missing contracts could tempt invented schemas | Abort; infer schema; degrade to verified presentation | R17 P05 | Continue with accepted design, label gap, keep handoff minimal | Delivers useful package without false authority | Some fields may need later refinement | J1 J2 J5 |
| Cross-template validation | Local correctness can still break lifecycle | Validate files alone; validate family boundaries and links | R05 R17 | Automated structure checks plus manual lifecycle review | Detects duplication and silent transitions | Cannot prove semantic correctness of every future fill | Package |
