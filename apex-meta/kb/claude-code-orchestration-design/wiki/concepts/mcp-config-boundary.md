# MCP Config Boundary

## Definition

The MCP config boundary is the rule that Model Context Protocol server configuration (project `.mcp.json`, user-level MCP settings, or managed-policy MCP entries) is a trust-sensitive runtime boundary that must be deliberately authored and approved by the operator, not silently generated by KB compilation. Phase 2 KB compile work may document, analyze, and defer MCP configuration policy, but it must never write or modify `.mcp.json` or equivalent runtime MCP config as a side effect of wiki synthesis.

## Operating Rules

```yaml
rules:
  - "Never write, generate, or modify .mcp.json or other MCP runtime configuration during Phase 2 KB compile (S6)."
  - "Treat MCP server scope (user, project, or managed) as a deliberate trust decision, not a default choice."
  - "Require explicit allowlist and operator approval before any MCP server configuration is committed to the repository."
  - "Document MCP config questions as deferred policy items, per operator decision Q005, rather than resolving them inside a KB compile step."
  - "Distinguish MCP config discussion (a wiki concept page) from MCP config action (writing or committing an actual .mcp.json), and never let the former imply the latter has happened."
```

## Macro / Meso / Micro

### Macro

Claude Code treats MCP server configuration as more consequential than an ordinary settings file because it determines what external systems the agent can reach and what external content it can be exposed to. Apex mirrors this by drawing a hard line around MCP config specifically: the KB can talk about MCP, reason about it, and file open questions about it, but the compile process itself is never the place where MCP config gets written or committed.

### Meso

This boundary sits alongside the broader soft-guidance-vs-hard-enforcement pattern in the KB: `.mcp.json` and MCP scope settings are configuration/enforcement-adjacent artifacts, not guidance prose, so changing them has direct operational effect (new tool surfaces, new external data access) rather than just shaping model behavior. Operator decision Q005 makes the deferral explicit — MCP policy and committed `.mcp.json` rules are pushed to a later phase, and the current Phase 2 treatment is to preserve the topic as an open question or a later policy page rather than to invent a policy now.

### Micro

B02-C013 grounds the underlying trust requirement: MCP connects Claude Code to external tools, databases, APIs, and event channels, but requires explicit trust evaluation because external content can expose prompt-injection risk (`phase1-batch02-claude-code-orchestration-surface.md` claim B02-C013). Operator decision Q005 (`operator-phase1-review-decisions-20260702.md` lines 86-89) states: "defer_mcp_policy_and_committed_mcp_json_rules," with `current_phase2_treatment: preserve_as_open_question_or_later_policy_page." The phase0 navigation report boundary (referenced in B02-C015's source pointers) reinforces that Phase 1/Phase 2 compile work must not perform Plan/Sync/Session or runtime writes outside its deterministic scope.
