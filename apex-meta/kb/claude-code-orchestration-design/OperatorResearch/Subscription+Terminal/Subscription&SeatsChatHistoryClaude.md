## Q2: How Does Claude Actually Measure Usage?

**"Messages per 5 hours" is a UI simplification.** Internally everything runs on tokens.[](https://www.truefoundry.com/blog/claude-code-limits-explained)

## The dual-layer quota system

Claude enforces two simultaneous controls:[](https://www.truefoundry.com/blog/claude-code-limits-explained)

1. **5-hour rolling window** — Starts on your first prompt. Caps burst activity. Resets 5 hours after that first prompt, not on a fixed clock.
    
2. **Weekly active compute hours cap** — Counts only time Claude is actively processing tokens (not idle time, browsing, pausing). Pro ≈ 40–80 Sonnet hours/week. Max tiers up to 480 Sonnet hours/week.[](https://www.truefoundry.com/blog/claude-code-limits-explained)
    

Both limits apply to the **same bucket** — Claude.ai chat, Claude Code, and Cowork all draw from the same quota. Hit either limit and all prompts block until reset.[](https://www.truefoundry.com/blog/claude-code-limits-explained)

## Why this matters for iterative small-step workflows

Token burn **does not scale linearly with message count** — it scales with **context**:[](https://www.truefoundry.com/blog/claude-code-limits-explained)

- Every turn in an agentic session replays: system prompt + full conversation history + all file references + tool definitions
    
- 5 medium files attached = 30,000+ tokens before you type a single word
    
- "Ultrathink" extended reasoning mode multiplies token cost ~5× vs. standard
    
- MCP servers and tools often consume more tokens than the actual prompt
    

**For your automation workflows** (event ticketing, billing, project management, venue finding): each tool-calling loop in an iterative workflow is expensive because the context window re-loads on every step. A 20-step iterative agentic workflow can burn a Team Standard seat's entire 5-hour window in one session.[](https://www.truefoundry.com/blog/claude-code-limits-explained)
