## Contract for `status-merge` (State Synchronizer core)

**Role:** deterministic aggregator producing a canonical registry over tasks and KB pages, suitable for all Sync‑cluster processes (PM4–PM5, PM7–PM8, KB4–KB5).

**Input (required):**

- A collection of entity records, each with at least:
    
    - `task_id` (for tasks) or a stable page identifier (for KB records).
        
    - `title`.
        
    - `status` ∈ {`pending`, `in_progress`, `blocked`, `done`} (H1).
        
    - `created_at`, `updated_at`.
        
    - Optional `priority`, `urgency`.
        
    - Optional `dependson`, `conflictswith`, `parallel`.
        
    - For KB pages: `category`, `sources`, `updated_at`.

**Processing responsibilities (LLM + optional Python helpers):**

- Read registry and session log, extract:
    
    - Tasks completed this session.
        
    - Tasks still `in_progress`.
        
    - Tasks `blocked` and their blocking dependencies.
        
    - Top N candidate tasks by `priority`, `urgency`, `unlock_depth` (using the numeric model, H7).
        
- Summarize key decisions and changes in narrative form, but avoid mutating underlying entity files directly (PD5’s operator gate principle).
    
- Package this into a structured Markdown handoff document.
    

**Output:**

- A single Markdown handoff file, e.g. `next-precap-context.md`, with:
    
    - YAML frontmatter including:
        
        - `last_session_at` (date).
            
        - Optional `focus_list` (list of `task_id`s preselected for next session).
            
    - Body sections (Markdown headings) such as:
        
        - “Summary of progress” — brief narrative based on `progress.md`/`log.md`.
            
        - “Current focus candidates” — ranked list with `task_id`, `title`, `status`, `priority`, `urgency`, `unlock_depth` and one‑sentence rationale each.
            
        - “Blockers and risks” — list of blocked items and causes.
            
        - “Suggested next actions” — explicit tasks for the next session, referencing canonical IDs.
