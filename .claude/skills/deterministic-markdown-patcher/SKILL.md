## Overview

This skill facilitates deterministic editing of Markdown and related
documentation.  Instead of writing diffs directly, the operator
provides a **patch intent** describing what to change.  The patch
executor then extracts the existing content from the live file, applies
the change exactly once and validates the diff before producing a result.
This approach prevents hallucinated modifications and ensures that
changes only occur where expected.  The skill is suitable for
updating documentation, front‑matter, Claude skill definitions or
patch packs in a reproducible manner.

## Inputs

The skill accepts the following inputs:

- **Patch intent**: A description of the desired change.  This can be a
  free‑form note or a structured JSON/YAML object (see the
  `patch‑intent.okf.md` template).  At minimum the intent must specify
  the target file and either an exact *old text* to replace or a
  heading name for section replacement.  For front‑matter edits the
  keys and values to set must be provided.
- **Target file**: The relative path to the Markdown or YAML file to be
  patched.  Only files within the allowed repository root may be
  modified.
- **Optional plan**: A patch plan generated by a previous invocation of
  this skill or by the operator.  Plans may aggregate multiple patch
  intents into a pack for batch execution.

## Outputs

Depending on the requested operation the skill produces one or more of
the following outputs:

- **Patched file**: The updated file with the requested change applied.
- **Unified diff**: A textual diff between the original and patched file
  used as proof of change.
- **Fixture file**: A JSON fixture describing the inputs, actions and
  expected outcome of the patch.  Fixtures can be re‑run using the
  `fixture_runner.py` script.
- **Final report**: A report summarising the actions taken, including
  validation results and any failure messages.

## Procedure

Follow these steps to perform a deterministic patch:

1. **Interpret the intent** – Read the operator’s patch intent and
   identify the target file, the old text or heading to be replaced,
   the new text, and the desired mode (replace‑once, replace‑heading‑
   section, front‑matter‑set, etc.).  Do not invent content: only
   describe the intent and candidate location.  See
   `patch‑intent.okf.md` for the expected structure.
2. **Inspect the file** – Use the `patch_executor.py` in `inspect`
   mode to read the live file and confirm that the old text or
   heading exists exactly once.  If not found or found multiple
   times, refine the intent or ask for clarification.
3. **Perform the replacement** – Invoke the appropriate mode on
   `patch_executor.py`:
   - *replace‑once* for an exact span replacement.
   - *replace‑heading‑section* for replacing an entire section under a
     heading.
   - *front‑matter‑set* for updating YAML front‑matter keys.
   The executor will extract the old span from the live file, apply
   the replacement once and write the patched file to disk.
4. **Validate the scope** – Run `patch_executor.py` with the
   `validate‑scope` mode to ensure that the changes are confined to the
   intended span or section.  If validation fails, do not continue
   until the intent has been corrected.
5. **Generate a diff** – Use the `diff` mode to produce a unified diff
   between the original and patched file.  This diff serves as proof
   of the change and should accompany the final report.
6. **Run fixtures (optional)** – When provided with a fixture, run
   `fixture_runner.py` to execute the patch and assert that the
   outcome matches the expected result.  Fix any discrepancies
   before proceeding.
7. **Record results** – Populate the final report template with the
   actions taken, the diff, and the validation status.  Include any
   failure modes encountered and how they were resolved.  Point to
   the relevant reference documents for more details.
8. **Handle failure modes** – If any failure mode arises (e.g., no
   matches, multiple matches, YAML parse errors), follow the
   recovery steps outlined in `failure‑modes‑and‑recovery.md`.

## Failure Modes

Common failure modes include:

- *No match*: The old text or heading is not found.  Verify the
  spelling, adjust context or inspect the file to locate the correct
  span.
- *Multiple matches*: The old text appears more than once.  Narrow
  the context or provide a more specific span so that exactly one
  match exists.
- *Ambiguous heading*: The heading appears at multiple levels or
  contexts.  Specify the full heading hierarchy (e.g., `## Parent
  heading › Subheading`).
- *YAML parse error*: Front‑matter cannot be parsed.  Ensure that the
  target file begins with a valid YAML front‑matter block (`---
  ... ---`).
- *Invalid path*: The target file is outside the allowed repository
  root.  Use only relative paths within the repository.
- *Validation failure*: The diff contains unexpected changes.  Run
  `validate‑scope` again and correct the intent.

See `failure‑modes‑and‑recovery.md` for detailed recovery strategies.

## Non‑Goals

This skill does **not**:

- Generate or apply arbitrary unified diffs by hand.
- Perform full‑file rewrites unless explicitly authorised via a
  generated‑file mode.
- Invent missing context when the old span cannot be found; the
  operator must supply sufficient detail.
- Edit files outside the repository root or modify binary files.
- Execute network requests or scripts other than the provided
  deterministic tools.

## Completion Gate

At the end of the patch process confirm that:

- [ ] The patch intent was interpreted correctly and required fields
  were identified.
- [ ] The specified old text or heading existed exactly once in the
  target file.
- [ ] The replacement was applied using the correct mode and only
  within the intended scope.
- [ ] Validation and diff generation succeeded without unexpected
  changes.
- [ ] A final report was produced summarising the actions and outcomes.

Only mark the task as complete when all boxes above are checked.  If
any step fails, refer to the failure modes and adjust the intent or
execution accordingly.
